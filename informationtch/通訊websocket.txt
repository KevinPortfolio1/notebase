## 第一階段：建立「長連線」的直覺

傳統 HTTP 是「你問，我答」，問完就斷開；
WebSocket 是「打電話」，電話接通後，雙方隨時都能說話。

### 1. 握手 (Handshake)
連線一開始會發送一個 HTTP 請求，帶有 `Upgrade: websocket` 標頭。
伺服器同意後，協定正式切換。
### 2. 事件驅動 (Event-driven)
你不需要用迴圈去檢查有沒有新訊息，而是編寫「事件處理器」：
* **onOpen**: 連線成功時要做什麼？（如：加入聊天室）
* **onMessage**: 收到資料時怎麼處理？（如：渲染畫面）
* **onClose**: 斷線時怎麼處理？（如：顯示離線狀態）
* **onError**: 發生錯誤時的報錯。

## 第二階段：實作最簡單的 Echo Server (Node.js 範例)

新手建議從 Node.js 的 `ws` 庫開始，因為它最貼近原生協定。

1. **安裝**: `npm install ws`
2. **伺服器端代碼**:
```javascript
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

wss.on('connection', (ws) => {
    console.log('新客到訪！');
    ws.on('message', (message) => {
        console.log(`收到訊息: ${message}`);
        ws.send(`伺服器回覆: 你說了 ${message}`);
    });
});
```

## 第三階段：從「練習」跨越到「實際應用」

在開發真實產品時，你會遇到以下三個痛點，這也是新手與資深開發者的分水嶺：

### 1. 心跳機制 (Heartbeat / Ping-Pong)
網路環境複雜（如移動網路、防火牆），連線可能會在沒有通知的情況下「死掉」。
* **做法**: 伺服器定期發送一個 Ping 給客戶端，若一段時間沒收到回覆 (Pong)，
就主動斷開連線釋放資源。

### 2. 斷線重連 (Auto Reconnect)
客戶端必須具備自動連線功能。
* **實務**: 使用「指數退避演算法」（第一次隔 1 秒重連，第二次 2 秒，第三次 4 秒...），
避免瞬間衝擊伺服器。

### 3. 身分驗證 (Authentication)
WebSocket 握手發生在 HTTP 階段，通常會在這裡檢查 **Token (JWT)**。
* **注意**: 握手成功後，該連線就是持續授權的狀態，直到斷線。

## 第四階段：生產環境的考量

當你的應用要上線給成千上萬人使用時：

* **負載平衡 (Load Balancing)**:
    普通的 Load Balancer 會隨機分配請求。
	但在 WebSocket 中，你需要 **"Sticky Sessions" (黏性會話)**，
	確保客戶端始終連到同一台伺服器，或者使用 **Redis Pub/Sub** 來同步多台伺服器間的訊息。
* **安全性 (WSS)**:
    永遠使用 `wss://` (WebSocket over TLS)，就像你永遠該用 HTTPS 一樣，
	這能防止中間人攻擊並提高防火牆穿透率。
* **併發限制**:
    單台伺服器能承載的連線數受限於記憶體與檔案描述符 (File Descriptors)，
	實際佈署時需要調整作業系統參數。
